Lab Database — Structure & Workflow

Big-picture map of the redesigned schema for FACS, ELISA, and histology data: what each table is for, how they connect, and how a measurement travels from a raw .wsp/.xlsx export to a queryable row.

The big picture

The schema splits cleanly into two layers. Dimensions describe the who/what/where of an experiment — they change rarely and are shared by everything downstream. Facts are the actual measurements — one table per assay domain, each shaped to match what that domain's import function already produces as a tidy long tibble.

DIMENSIONS

domains · experiments · subjects · samples · sample_digestions · assays

FACTS

facs_stains · facs_measurements · elisa_measurements · histo_measurements

From raw file to analysis workflow

Every domain follows the same pipeline: a raw export is parsed by an existing package function into a tidy long tibble, written into the matching fact table behind a UNIQUE constraint (so re-running an import is idempotent, not additive), then queried back out across experiments through the same dplyr::tbl() interface already in use today.

Why this shape

  • idempotent importsEvery fact table's UNIQUE constraint means re-running an import script can't silently duplicate rows — DuckDB rejects the collision instead of the script relying on a manual check-then-append.
  • facs_stains splitVolumes that vary per stained panel (vol_stained, vol_resuspended/vol_measured or the bead-counting equivalents) live one level below the sample, since one digested tissue sample is routinely split into several differently-stained tubes.
  • count_method branchfacs_stains carries both volumetric (HTS) and bead-based counting fields, with a CHECK constraint enforcing only the fields matching the chosen method are filled in.
  • domains lookupAssay domains are rows in a table, not a hard-coded list — adding scRNA-seq or any future modality later is an INSERT, not a schema migration.
  • provenanceEvery measurement row carries source_file back to the raw export it came from — the database never loses the thread to the original data.
Out of scope here: the R-facing db_ functions that will read/write this schema, and migrating the existing MPOmRNA.db data into it, are separate design work that follows once this structure is locked in.
